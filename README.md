### CPU의 작동 원리

#### ALU와 제어장치

**ALU (산술논리연산장치)**

ALU는 레지스터로부터 피연산자를 받아들이고, 제어장치로부터 수행할 연산을 알려주는 제어 신호를 받아들인다. ALU는 연산을 수행한 뒤 **결괏값**과 **플래그**를 내보낸다.

- 결괏값은 레지스터에 일시적으로 저장됨 (메모리 접근보다 레지스터 접근이 빠르기 때문)
- 연산 결과에 대한 부가 정보는 **플래그 레지스터**에 저장됨

**플래그(Flag)**

ALU가 연산 결과에 대해 내보내는 부가 정보이다. 플래그는 **플래그 레지스터**에 저장된다.

| 플래그 | 의미 | 설명 |
|--------|------|------|
| 부호 플래그 | 연산 결과의 부호 | 1이면 음수, 0이면 양수 |
| 제로 플래그 | 연산 결과가 0인지 여부 | 1이면 결과가 0 |
| 캐리 플래그 | 올림수 또는 빌림수 발생 여부 | 1이면 올림/빌림 발생 |
| 오버플로우 플래그 | 오버플로우 발생 여부 | 1이면 오버플로우 발생 |
| 인터럽트 플래그 | 인터럽트 가능 여부 | 1이면 인터럽트 가능 |
| 슈퍼바이저 플래그 | 커널 모드 / 사용자 모드 | 1이면 커널 모드 |

**제어장치(CU, Control Unit)**

제어장치는 **제어 신호**를 내보내고 **명령어를 해석**하는 장치이다. 제어장치가 받아들이는 정보와 내보내는 정보는 다음과 같다.

제어장치가 **받아들이는** 정보:
- **클럭 신호**: 컴퓨터의 모든 부품을 일사불란하게 움직이게 하는 시간 단위
- **해석할 명령어**: 명령어 레지스터에 저장된 명령어를 받아들여 해석하고, 필요한 제어 신호를 발생시킴
- **플래그 레지스터 값**: 플래그 값을 참고하여 제어 신호를 발생시킴
- **외부로부터의 제어 신호**: 시스템 버스(제어 버스)를 통해 전달된 제어 신호를 받아들임

제어장치가 **내보내는** 제어 신호:
- **CPU 내부로**: ALU에 수행할 연산을 지시하거나, 레지스터 간 데이터 이동을 지시
- **CPU 외부로**: 메모리에 읽기/쓰기를 지시하거나, 입출력장치에 읽기/쓰기를 지시

#### 레지스터

레지스터는 CPU 내부의 작은 임시 저장 장치로, 프로그램 실행에 필요한 값들을 빠르게 저장하고 접근하는 데 사용된다. 프로그램 속 명령어와 데이터는 실행 전후로 레지스터에 저장되므로, 레지스터의 값을 관찰하면 프로그램의 실행 흐름을 파악할 수 있다.

**프로그램 카운터 (PC, Program Counter)**
- **다음에 메모리에서 가져올 명령어의 주소**를 저장하는 레지스터
- 명령어를 가져올 때마다 자동으로 증가하여 다음 명령어를 가리킴
- **명령어 포인터(IP, Instruction Pointer)**라고도 불림

**명령어 레지스터 (IR, Instruction Register)**
- **현재 실행 중인 명령어(방금 메모리에서 가져온 명령어)**를 저장하는 레지스터
- 제어장치가 이 레지스터의 명령어를 해석하여 제어 신호를 발생시킴

**메모리 주소 레지스터 (MAR, Memory Address Register)**
- **메모리에 접근할 주소**를 저장하는 레지스터
- CPU가 읽고자 하는 주소값을 주소 버스로 내보내기 위해 사용

**메모리 버퍼 레지스터 (MBR, Memory Buffer Register)**
- **메모리와 주고받을 값(데이터 또는 명령어)**을 저장하는 레지스터
- 데이터 버스로 주고받을 값이 이 레지스터를 거침
- **메모리 데이터 레지스터(MDR, Memory Data Register)**라고도 불림

**범용 레지스터 (General Purpose Register)**
- **데이터와 주소를 모두 저장**할 수 있는 레지스터
- 일반적인 상황에서 자유롭게 사용 가능
- CPU 안에 **여러 개**가 존재함

**플래그 레지스터 (Flag Register)**
- ALU 연산 결과에 따른 **부가 정보(플래그)**를 저장하는 레지스터
- 연산 결과의 부호, 제로 여부, 캐리, 오버플로우 등의 정보를 비트 단위로 저장

**명령어가 실행되는 과정 예시 (레지스터 관점)**

```
1. [PC] = 100번지  →  메모리 100번지의 명령어를 가져와야 함
2. [MAR] ← 100     →  주소 버스를 통해 100번지를 메모리에 전달
3. [MBR] ← 메모리[100]  →  메모리에서 명령어를 데이터 버스로 받아옴
4. [IR] ← [MBR]    →  가져온 명령어를 명령어 레지스터에 저장
5. [PC] ← 101      →  프로그램 카운터 증가 (다음 명령어를 가리킴)
6. 제어장치가 [IR]의 명령어를 해석하고 제어 신호를 발생
```

#### 스택 주소 지정 방식과 변위 주소 지정 방식

**스택 주소 지정 방식 (Stack Addressing)**
- **스택**과 **스택 포인터**를 이용한 주소 지정 방식
- 스택 포인터: 스택의 꼭대기(Top)를 가리키는 레지스터
- 오퍼랜드 없이 스택의 최상위 데이터를 대상으로 연산 (0-주소 명령어에서 사용)
- 메모리 안에 스택 영역이 존재하며, 스택 포인터가 해당 영역의 어느 위치까지 데이터가 채워져 있는지를 가리킴

```
┌─────────────┐
│   데이터 C   │  ← 스택 포인터(SP)가 가리키는 위치
├─────────────┤
│   데이터 B   │
├─────────────┤
│   데이터 A   │
└─────────────┘
   메모리 스택 영역
```

**변위 주소 지정 방식 (Displacement Addressing)**
- 오퍼랜드 필드의 값과 **특정 레지스터의 값을 더하여** 유효 주소를 구하는 방식
- 명령어 구조: `연산 코드 + 레지스터 + 오퍼랜드`

```
┌──────────┬──────────┬──────────┐
│ 연산 코드 │ 레지스터  │ 오퍼랜드  │
└──────────┴──────────┴──────────┘
                │           │
                └─────┬─────┘
                  더하기(+)
                      │
                      ▼
                 유효 주소
```

변위 주소 지정 방식은 더하는 레지스터에 따라 **상대 주소 지정 방식**과 **베이스 레지스터 주소 지정 방식**으로 나뉜다.

**상대 주소 지정 방식 (Relative Addressing)**
- 오퍼랜드 필드의 값 + **프로그램 카운터(PC)** = 유효 주소
- "현재 실행 위치로부터 얼마나 떨어진 주소에 있는가"를 나타냄
- 분기(조건/무조건 점프)에 주로 사용

```
유효 주소 = PC + 오퍼랜드

예) PC = 100, 오퍼랜드 = +3  →  유효 주소 = 103 (세 칸 앞으로 점프)
예) PC = 100, 오퍼랜드 = -5  →  유효 주소 = 95  (다섯 칸 뒤로 점프)
```

**베이스 레지스터 주소 지정 방식 (Base Register Addressing)**
- 오퍼랜드 필드의 값 + **베이스 레지스터** = 유효 주소
- 베이스 레지스터에는 기준 주소(base address)가 저장되어 있고, 오퍼랜드는 **기준 주소로부터의 떨어진 거리(변위, offset)**를 나타냄

```
유효 주소 = 베이스 레지스터 + 오퍼랜드

예) 베이스 레지스터 = 200, 오퍼랜드 = 50  →  유효 주소 = 250
```

### 명령어 사이클과 인터럽트

#### 명령어 사이클 (Instruction Cycle)

CPU가 하나의 명령어를 처리하는 과정을 **명령어 사이클**이라 하며, 프로그램이 실행되는 동안 이 사이클이 반복된다.

- **인출 사이클(Fetch Cycle)**: 메모리에서 명령어를 가져오는 단계
- **실행 사이클(Execute Cycle)**: 가져온 명령어를 해석하고 실행하는 단계
- **간접 사이클(Indirect Cycle)**: 간접 주소 지정 방식을 사용하는 경우, 유효 주소를 가져오기 위해 메모리에 한 번 더 접근하는 단계

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 인출 사이클│ →  │간접 사이클│ →  │ 실행 사이클│
│ (Fetch)  │    │(Indirect)│    │(Execute) │
└──────────┘    └──────────┘    └──────────┘
     ↑               ↑                │
     │    (간접 주소일 때만)             │
     └─────────────────────────────────┘
```

#### 인터럽트 (Interrupt)

CPU의 작업을 방해하는 신호를 **인터럽트**라 한다. 인터럽트는 발생 원인에 따라 **동기 인터럽트**와 **비동기 인터럽트**로 나뉜다.

**동기 인터럽트 (Synchronous Interrupt) = 예외(Exception)**
- CPU가 명령어를 실행하는 도중 **CPU 내부에서 발생**하는 인터럽트
- 프로그래밍 오류 등 예외적인 상황에서 발생
- 예외의 종류:
  - **폴트(Fault)**: 예외 처리 후 **예외가 발생한 명령어부터** 다시 실행 (예: 페이지 폴트)
  - **트랩(Trap)**: 예외 처리 후 **다음 명령어부터** 실행 (예: 디버깅 시 사용하는 브레이크포인트)
  - **중단(Abort)**: CPU가 실행을 계속할 수 없는 심각한 오류로, 프로세스를 강제 중단
  - **소프트웨어 인터럽트**: 시스템 호출(System Call)처럼 의도적으로 발생시키는 인터럽트

**비동기 인터럽트 (Asynchronous Interrupt) = 하드웨어 인터럽트**
- 주로 **입출력장치 등 CPU 외부**에서 발생하는 인터럽트
- CPU가 입출력 작업 완료 등 외부 알림을 받을 때 사용
- CPU는 입출력 작업 동안 다른 일을 처리하다가, 하드웨어 인터럽트를 받으면 해당 작업을 처리

**하드웨어 인터럽트 처리 순서**

```
1. 입출력장치가 CPU에 인터럽트 요청 신호를 보냄
2. CPU는 실행 사이클이 끝난 후 인터럽트 여부를 확인
3. 인터럽트 플래그를 통해 현재 인터럽트를 받을 수 있는지 확인
4. 인터럽트를 받을 수 있다면, 현재 작업 내용을 스택에 백업 (복귀 주소 등)
5. 인터럽트 벡터를 참조하여 해당 인터럽트 서비스 루틴(ISR)의 시작 주소를 찾음
6. 인터럽트 서비스 루틴(ISR)을 실행하여 인터럽트를 처리
7. ISR 실행이 끝나면, 스택에 백업해 둔 작업 내용을 복원하여 원래 작업으로 복귀
```

- **인터럽트 요청 신호**: 인터럽트를 요청하기 위해 보내는 신호
- **인터럽트 플래그**: 인터럽트 수용 가능 여부를 나타내는 플래그 (단, 정전이나 하드웨어 고장 등 무시할 수 없는 인터럽트도 존재)
- **인터럽트 벡터**: 인터럽트 서비스 루틴의 시작 주소를 포함하는 정보
- **인터럽트 서비스 루틴(ISR, Interrupt Service Routine)**: 인터럽트를 처리하기 위해 실행되는 프로그램. **인터럽트 핸들러**라고도 불림

### CPU 성능 향상 기법

#### 빠른 CPU를 위한 설계 기법

**클럭 (Clock)**
- 컴퓨터의 모든 부품이 동기화되어 작동하기 위한 **시간 단위**
- **클럭 속도**: 1초에 클럭이 반복되는 횟수. 단위는 **Hz(헤르츠)**
  - 예: 클럭 속도가 2.5GHz → 1초에 클럭이 25억 번 반복
- 클럭 속도가 높을수록 CPU는 더 빠르게 동작하지만, 무작정 높이면 **발열** 문제가 발생
- 따라서 클럭 속도만으로는 성능 향상에 한계가 있으며, 다른 방법도 함께 고려해야 함

**코어와 멀티코어**
- **코어(Core)**: 명령어를 실행하는 부품들의 집합. 과거에는 CPU 안에 코어가 하나뿐이었지만, 현대 CPU에는 **여러 개의 코어**가 존재
- **멀티코어 CPU(멀티코어 프로세서)**: 코어를 여러 개 포함하고 있는 CPU
  - 듀얼코어(2개), 쿼드코어(4개), 옥타코어(8개) 등
- 코어 수가 많아지면 이론상 처리 속도가 비례하여 증가하지만, 실제로는 코어마다 처리할 연산이 적절히 분배되어야 하므로 코어 수에 정비례하여 성능이 향상되지는 않음

**스레드와 멀티스레드**

스레드(Thread)라는 용어는 하드웨어와 소프트웨어에서 의미가 다르다.

**하드웨어적 스레드 (Hardware Thread)**
- 하나의 코어가 **동시에 처리할 수 있는 명령어 단위**
- 1코어 1스레드 CPU: 하나의 코어에서 한 번에 하나의 명령어를 처리
- **멀티스레드 프로세서(Multithread Processor)**: 하나의 코어에서 여러 개의 명령어를 동시에 처리할 수 있는 CPU
  - 예: 2코어 4스레드 CPU → 한 번에 4개의 명령어를 동시 처리 가능
- 하드웨어 스레드가 여러 개인 CPU는 프로그램 입장에서 여러 개의 CPU처럼 보이므로, 이를 **논리 프로세서(Logical Processor)**라고 부름
- 멀티스레드를 구현하려면 레지스터 세트(프로그램 카운터, 스택 포인터, 범용 레지스터 등)가 **하드웨어 스레드 수만큼** 존재해야 함

**소프트웨어적 스레드 (Software Thread)**
- 하나의 프로그램에서 **독립적으로 실행되는 단위**
- 하나의 프로그램이 여러 기능을 동시에 수행할 때 각 기능이 소프트웨어 스레드에 해당
- 예: 워드 프로세서에서 입력을 받으면서 동시에 맞춤법 검사를 수행

#### 명령어 병렬 처리 기법

**명령어 파이프라인 (Instruction Pipeline)**

명령어 처리 과정을 여러 단계로 나누고, 각 단계를 **동시에(겹쳐서)** 수행하는 기법이다. 공장의 조립 라인과 유사한 개념이다.

명령어 처리의 대표적인 4단계:
1. **명령어 인출(IF, Instruction Fetch)**
2. **명령어 해석(ID, Instruction Decode)**
3. **명령어 실행(EX, Execute)**
4. **결과 저장(WB, Write Back)**

```
시간 →   t1     t2     t3     t4     t5     t6     t7
명령어1  [IF]   [ID]   [EX]   [WB]
명령어2         [IF]   [ID]   [EX]   [WB]
명령어3                [IF]   [ID]   [EX]   [WB]
명령어4                       [IF]   [ID]   [EX]   [WB]
```

파이프라이닝 없이 처리하면 16클럭이 필요하지만, 파이프라이닝을 적용하면 7클럭만에 4개의 명령어를 처리할 수 있다.

하지만 특정 상황에서는 파이프라인이 정상적으로 작동하지 않는 경우가 있으며, 이를 **파이프라인 위험(Pipeline Hazard)**이라 한다.

**데이터 위험 (Data Hazard)**
- 명령어 간 **데이터 의존성**으로 인해 발생하는 위험
- 이전 명령어의 결과를 다음 명령어가 필요로 하는데, 아직 결과가 준비되지 않은 경우

```
명령어 1: R1 ← R2 + R3    (R1에 값을 저장)
명령어 2: R4 ← R1 + R5    (R1의 값이 필요 → 명령어 1이 끝나야 사용 가능)
```

**제어 위험 (Control Hazard)**
- **분기(Jump, Branch)** 등으로 인해 프로그램 카운터가 갑자기 변경되어 미리 가져온 명령어들이 쓸모없어지는 위험
- 파이프라인에 미리 가져와 둔(인출한) 명령어가 실행되지 않아야 할 명령어인 경우
- 해결 방법 중 하나: **분기 예측(Branch Prediction)** — 프로그램이 어디로 분기할지 미리 예측하여 해당 명령어를 인출

**구조적 위험 (Structural Hazard) = 자원 위험(Resource Hazard)**
- 서로 다른 명령어가 **동시에 같은 CPU 부품(자원)**을 사용하려 할 때 발생하는 위험
- 예: 두 명령어가 동시에 메모리에 접근하려 하는 경우

**슈퍼스칼라 (Superscalar)**
- **여러 개의 파이프라인을 동시에** 사용하여 명령어를 처리하는 구조
- CPU 내부에 여러 개의 명령어 파이프라인을 두어, 매 클럭 사이클마다 여러 명령어를 동시에 인출·해석·실행 가능

```
파이프라인 A:  [IF] [ID] [EX] [WB]
파이프라인 B:  [IF] [ID] [EX] [WB]
→ 매 클럭 사이클마다 2개의 명령어를 동시에 처리
```

- 파이프라인 위험의 가능성이 높아지므로, 이를 효과적으로 관리하는 것이 중요

**비순차적 명령어 처리 (OoOE, Out-of-Order Execution)**
- 명령어를 프로그램에 작성된 **순서대로가 아니라**, 데이터 의존성이 없는 명령어를 먼저 처리하여 파이프라인 효율을 높이는 기법
- 순서를 바꿔도 **프로그램의 실행 결과에 영향이 없는** 경우에만 수행

```
원래 순서:                     비순차 실행:
1. R1 ← MEM[100]              1. R1 ← MEM[100]
2. R2 ← R1 + R3  (1번 대기)    2. R4 ← R5 + R6  (의존성 없으므로 먼저 실행)
3. R4 ← R5 + R6               3. R2 ← R1 + R3  (1번 완료 후 실행)
→ 2번이 1번을 기다리는 동안 3번을 먼저 실행하여 시간 절약
```

#### CISC와 RISC

**명령어 집합 (ISA, Instruction Set Architecture)**
- CPU가 이해할 수 있는 명령어들의 모음
- CPU마다 명령어 집합이 다를 수 있으며, 이는 곧 CPU의 언어가 다르다는 것을 의미
- 명령어 집합의 설계 철학에 따라 **CISC**와 **RISC** 두 가지 방식으로 크게 나뉨

**CISC (Complex Instruction Set Computer)**
- **복잡하고 다양한 명령어**를 사용하는 CPU 설계 방식
- 대표: x86, x86-64 아키텍처 (Intel, AMD)

특징:
- 명령어의 형태와 크기가 **가변적** (명령어 길이가 제각각)
- 하나의 명령어로 복잡한 작업을 수행 가능 → **적은 수의 명령어로 프로그램 작성 가능**
- 다양한 주소 지정 방식을 지원
- 명령어의 실행 시간이 명령어마다 다름 (1클럭 ~ 수십 클럭)
- 명령어가 복잡하고 실행 시간이 일정하지 않아 **파이프라이닝에 불리**
- 전체 명령어 중 자주 사용되는 명령어는 일부에 불과

**RISC (Reduced Instruction Set Computer)**
- **짧고 규격화된 명령어**를 사용하는 CPU 설계 방식
- 대표: ARM 아키텍처 (스마트폰, 태블릿, 애플 M 시리즈 등)

특징:
- 명령어의 형태와 크기가 **고정적** → 명령어 해석이 단순
- 명령어 수가 적고, 명령어당 실행 시간이 **1클럭** 내외로 일정
- 파이프라이닝에 **최적화**되어 있음
- 메모리 접근은 `LOAD`와 `STORE` 두 명령어로만 수행 (load-store 구조)
- 범용 레지스터의 수가 많아 레지스터를 적극적으로 활용
- CISC보다 명령어 수가 많아질 수 있지만, 파이프라인 효율이 좋아 전체 성능은 향상

**CISC vs RISC 비교 요약**

| 항목 | CISC | RISC |
|------|------|------|
| 명령어 복잡도 | 복잡하고 다양 | 단순하고 적음 |
| 명령어 길이 | 가변적 | 고정적 |
| 명령어 실행 시간 | 명령어마다 다름 | 1클럭 내외로 일정 |
| 주소 지정 방식 | 다양함 | 적음 |
| 메모리 접근 | 다양한 명령어로 가능 | LOAD / STORE만 가능 |
| 레지스터 수 | 적음 | 많음 |
| 파이프라이닝 | 불리 | 유리 |
| 프로그램 길이 | 짧음 (명령어가 강력) | 김 (단순한 명령어 조합) |
| 대표 아키텍처 | x86, x86-64 | ARM, MIPS, RISC-V |
